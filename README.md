### Graduate Students

1. In the ISS sample application, data is persisted in DynamoDB. If this were a much higher-frequency application (hundreds of writes per minute), what changes would you make to the persistence strategy and why?

   **Answer:** The current design issues a single synchronous `put_item()` call once every 15 minutes from a CronJob, writing to a single partition keyed on `satellite_id = "ISS"`. At hundreds of writes per minute that design breaks down in two related ways. First, every write lands on the same DynamoDB partition, creating a hot partition that gets throttled once it approaches the per-partition write limit (~1,000 WCUs/second); second, the CronJob model itself cannot produce hundreds of writes per minute—you would need a continuously running producer such as a Deployment or a Kinesis/Kafka stream consumer. To address the hot partition, I would add a write-shard suffix to the partition key (e.g., `ISS#0` through `ISS#9`, chosen by `hash(timestamp) % 10`), distributing writes across ten logical partitions and multiplying effective throughput tenfold; reads would then fan out across all shards and merge results in the application layer. I would also switch from individual `put_item()` calls to `batch_writer()` so up to 25 items are flushed per API round-trip, cutting per-item latency and cost. For very high sustained rates I would front DynamoDB with Amazon Kinesis Data Streams, buffer incoming position records, and use a Lambda consumer to aggregate them before writing, reducing write amplification. I would also introduce a TTL attribute so records older than a configurable retention window are automatically expired by DynamoDB, keeping partition size bounded and avoiding the unbounded `fetch_history()` full-table scan that currently runs on every execution—at high frequency that scan would eventually read thousands of pages and become the dominant cost driver.

2. The ISS tracker detects orbital burns by comparing consecutive altitude readings. Describe at least one way this detection logic could produce a false positive, and how you would make it more robust.

   **Answer:** The detection logic in `altitude_analysis()` (app.py line 83) fires an `ORBITAL_BURN` label whenever a single sample's altitude is ≥ 1.0 km higher than the immediately preceding stored sample. The most straightforward source of false positives is a transient bad reading from the upstream `wheretheiss.at` API: if the API returns a glitched altitude—say, 410 km instead of 408 km due to a momentary interpolation error or stale cached value—the application will store that value, flag it as a burn, and then on the next execution the delta will swing sharply negative, producing a phantom spike with no corresponding real maneuver. A second, subtler false positive arises when CronJob executions are missed: if several consecutive runs fail (network issues, pod eviction, ImagePullBackOff), the "previous" record in DynamoDB could be an hour or more old. Over that gap, natural orbital oscillation and measurement drift can accumulate enough apparent altitude change to cross the 1.0 km threshold even without any engine firing. To make detection more robust I would require at least two or three consecutive readings above the threshold before emitting `ORBITAL_BURN`—a simple debounce that a single anomalous sample cannot satisfy. I would also add a plausibility guard: if the current altitude falls outside the ISS's known operational band (~390–425 km) it should be discarded before comparison. Finally, I would track the age of the previous record and mark the delta as `UNCERTAIN` rather than trusting it when the gap between readings exceeds a configurable maximum (e.g., 20 minutes), preventing long-gap comparisons from masquerading as burns.

3. How does each `CronJob` pod get AWS permissions without credentials being passed into the container?

   **The EC2 instance that hosts the Kubernetes node has an IAM instance profile (instance role) attached to it at the AWS layer.** When the Python process inside the pod calls `boto3.resource("dynamodb", ...)` without any explicit credentials, boto3 works through its standard credential-provider chain. It finds nothing in environment variables (no `AWS_ACCESS_KEY_ID` is set in the manifest), nothing in `~/.aws/credentials` inside the container, and eventually reaches the EC2 Instance Metadata Service (IMDS), a link-local HTTP endpoint reachable at `169.254.169.254` from any process running on the instance. IMDS returns a set of short-lived STS credentials—access key ID, secret access key, and session token—that are automatically rotated every few hours by AWS. These credentials are fetched into memory by boto3, used for the duration of the API calls, and never written to any file, which is why no secrets appear in the manifest, the image, or any volume. The production-grade evolution of this pattern is IAM Roles for Service Accounts (IRSA): the EKS cluster is configured with an OIDC identity provider, a Kubernetes `ServiceAccount` is annotated with an IAM role ARN, and EKS injects a signed OIDC token into the pod at a well-known path; boto3's `web_identity` credential provider then exchanges that token with AWS STS for temporary credentials scoped to the specific pod, giving per-workload credential isolation rather than the per-node isolation that the instance-profile approach provides.

4. Notice the structure of the `iss-tracking` table in DynamoDB. What is the partition key and what is the sort key? Why do these work well in this example, but may not work for other solutions?

   **The partition key is `satellite_id` (a string, always `"ISS"`) and the sort key is `timestamp` (a string in ISO 8601 UTC format, e.g., `"2024-04-09T12:34:56Z"`).** This combination works well for the current application for several reasons: there is exactly one entity being tracked, so every item belongs to the same partition and a single key-condition query (`satellite_id = "ISS"`) retrieves the complete history without a table scan; ISO 8601 strings sort lexicographically in the same order as chronologically, so `ScanIndexForward=False, Limit=1` efficiently finds the most recent record without reading unnecessary items; and because each CronJob execution fires at a unique wall-clock time, the sort key is naturally unique within the partition, so there is no risk of `put_item()` silently overwriting a different item. However, these choices do not generalize well. If you were tracking hundreds of satellites the partition key would still fan out evenly—one partition per `satellite_id`—which is fine, but all write traffic for a heavily polled satellite would still concentrate on a single partition and could be throttled. More critically, `satellite_id = "ISS"` is always the same value, so there is only ever one partition for this table; at higher write rates that single partition becomes a bottleneck (see question 1). The string-typed timestamp sort key is also a subtle limitation: because DynamoDB stores it as a string rather than a Number, you cannot perform arithmetic range expressions against it natively—computing "the last 30 minutes of readings" requires the caller to format timestamp strings rather than doing simple numeric subtraction. Storing epoch milliseconds as a Number sort key would be more flexible. Finally, the design assumes a single producer; in a distributed multi-process scenario two writers could share the exact same second-resolution timestamp and the second `put_item()` would silently overwrite the first—a composite sort key such as `timestamp#uuid` would eliminate that collision risk.
